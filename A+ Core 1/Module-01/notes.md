# Module 1 – What Does an IT Specialist Do?

## 1.1 The Role of an IT Specialist

### Topics Covered
- Role of an IT specialist
- Skills and abilities required
- Troubleshooting methodology
- Documentation and escalation procedures

### Key Responsibilities
An IT specialist is responsible for:
- Troubleshooting hardware and software issues
- Assisting users with technical problems
- Documenting findings and solutions
- Escalating issues when necessary
- Maintaining systems and infrastructure

### Important Skills
- Problem-solving
- Communication
- Attention to detail
- Patience
- Research abilities
- Documentation skills
- Logical thinking

---

# 1.2 Troubleshooting Methodology

The CompTIA troubleshooting methodology follows a structured process.

---

## Step 1 – Identify the Problem

### Best Practices
- Gather information from the user
- Identify recent changes
- Determine whether backups exist before making changes
- Begin documenting immediately
- Ask about environmental or infrastructure changes

### Useful Questions to Ask
- What exact error message appears?
- Are there unusual sounds or symptoms?
- Is anyone else experiencing the issue?
- When did the problem start?
- Were any recent changes made?
- Did the issue work previously?
- Has the user already tried anything?

### Physical Inspection
Look and listen for:
- Scorched motherboard
- Clicking or “sick” hard drive sounds
- Fan noise issues
- Loose cables or connectors

### Research Sources
- Internal documentation
- Event logs
- Diagnostic tools
- Vendor documentation
- Support forums
- Knowledge bases
- Other technicians or SMEs (Subject Matter Experts)

---

## Step 2 – Establish a Theory of Probable Cause

### “Question the Obvious”
Sometimes simple problems cause major symptoms.

### Troubleshooting Tips
- Reproduce the problem if possible
- Observe the user performing the task
- Break systems into smaller subsystems
- Eliminate non-causes systematically

### Determine Whether the Issue Is:
- Hardware-related
- Software-related
- Network-related
- User-related

### Examples
- Faulty RAM
- Power supply failure
- Bad cables/connectors
- Corrupted software
- File system issues

### Conduct Research
IT technicians must learn to:
- Use search engines effectively
- Search databases and documentation
- Locate relevant technical information quickly

---

## Step 3 – Test the Theory

After forming a theory:
- Perform tests to confirm the cause
- If confirmed → continue to solution
- If not confirmed → establish a new theory or escalate

### Escalation
If the issue cannot be solved efficiently:
- Escalate to senior technicians
- Contact vendors or manufacturers
- Use support contracts or warranty services

### Important Reminder
Never share confidential or proprietary information publicly when seeking help online.

---

## Step 4 – Establish a Plan of Action

### Common Solutions
1. Repair  
2. Replace  
3. Workaround  

### Considerations Before Acting
- Cost
- Time
- Resources required
- Impact on users/business
- Authorization and company policies

### Additional Concepts
- Rollback changes if necessary
- Test after each change
- Schedule disruptive fixes appropriately
- Follow vendor instructions carefully

### RMA
Returning faulty hardware under warranty usually requires:
- **RMA (Returned Material Authorization)**

---

## Step 5 – Verify Full System Functionality

### Verification Tasks
- Confirm the original issue is resolved
- Test overall system functionality
- Check logs and indicators
- Inspect hardware connections
- Update software or drivers if necessary

### Preventive Measures
Implement changes to help prevent the issue from happening again.

---

## Step 6 – Document Findings, Actions, and Outcomes

Documentation is extremely important in IT.

### Documentation Should Include
- Findings
- Actions taken
- Lessons learned
- Final outcome
- Preventive measures

### Why Documentation Matters
- Helps future technicians
- Builds a knowledge base
- Supports ticket systems
- Tracks recurring problems
- Demonstrates value to the business
- Improves troubleshooting efficiency

### Ticketing Systems
Troubleshooting often happens within ticket systems that:
- Track responsibility
- Record actions
- Store solutions
- Help identify recurring issues

---

# Key Takeaways

- Troubleshooting should always follow a structured process.
- Documentation is just as important as solving the problem.
- IT support requires both technical and communication skills.
- Escalation is part of the job, not a failure.
- Research and continuous learning are essential in IT.
- Preventive measures can reduce future incidents.
